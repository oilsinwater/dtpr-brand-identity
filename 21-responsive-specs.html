Responsive Application Specs

DTPR operates across an unusual range of contexts — from 10-meter viewing distances at outdoor installations to 320px mobile screens accessed via QR scan seconds after encountering a physical sign. This document specifies how the visual system transforms across that full range, following the Process Box transformation rules from the three Sprint Recipes.

Physical Viewing Distance — Scale Rules (Regulatory Track)

Physical signage operates across three standard viewing zones. Each zone has a distinct minimum readable element size derived from the hexagon blueprint formula.

Awareness Zone (10m+)

Designed to capture attention from a distance in public environments. In this zone, citizens can identify the presence of DTPR signs, count the active categories, and see primary silhouettes.

Minimum Hexagon Size: R = 90mm (Width = 180mm)

Engagement Zone (1–2m)

The standard reading distance for pedestrians. Icon labels, category descriptions, DTPR taxonomy headings, and primary QR code targets are clearly visible and legible.

Minimum Hexagon Size: R = 36mm (Width = 72mm)

Scan / Read Zone (0.3–0.5m)

Interactive touch and scan distance. Citizens are actively reading footer details, purpose statements, data retention specifications, and scanning the QR code using mobile viewports.

Minimum Hexagon Size: R = 22mm (Standard Panel)

Transformation Rules by Sprint Recipe

Each recipe defines its own transformation envelope — the specific rules for how form elements respond to context. These are not just breakpoints; they are semantic shifts in how the system presents itself.

Recipe A — Regulatory Track: Strict Cascade

Regulatory contexts require predictability above flexibility. The grid never breaks — it compresses to accommodate fewer items, not to create new layouts.

  • ≥ 1280pxFull 3-column doc layout. Persistent sidebar. All hex icons at 32px+. Full footer with operator metadata.
  • 768–1279px2-column layout (sidebar collapses to top nav). Hex icons at 24px. Footer remains full.
  • 428–767pxSingle column. Sidebar becomes accordion. Hex icons at 20px. QR code moves above footer.
  • ≤ 427pxSingle column. No sidebar. Hex icons at 16px. Body text 16px minimum. All non-essential chrome removed. "Survival mode" — only data disclosure and feedback CTA visible.

Recipe B — Maker Track: Fluid Overlay

Developer and taxonomy contexts prioritize content density and interaction. The grid expands to fill available space; sidebar is always present on 768px+.

  • ≥ 1280pxFull taxonomy explorer. 4+ hex columns. Filter bar always visible. Detail panel right-rail. Code blocks full-width.
  • 768–1279pxLeft sidebar + 3-column hex grid. Detail panel slides up from bottom. Code blocks responsive-scroll.
  • 428–767pxNo sidebar (filter becomes dropdown). 2-column hex grid. Detail panel full-screen overlay on tap.
  • ≤ 427px1-column hex list (not grid). Tap to expand detail. Code blocks horizontal-scroll. Graph-paper texture removed.

Recipe C — Commons Track: Progressive Disclosure

Citizen-facing contexts operate mobile-first by definition (QR scan arrivals). Desktop is never the primary context — desktop views should redirect to the deployer or developer entry.

  • ≤ 427pxPrimary context. Single column. One hex icon prominent. Plain-language disclosure. Rating CTA above fold. No nav chrome.
  • 428–767px2-hex row possible. Expand to show second technology type. Feedback form inline.
  • ≥ 768pxRedirect notice: "You're on a large screen. This page is designed for mobile QR-scan access. Are you a deployer or developer?" → route to correct persona entry.

Dynamic Visual Identity — Transformation Summary

  1. Transformation ≠ just breakpoints: In dynamic visual identity terms, the Process Box is transformation-based vocabulary — identification through a distinctive process, not a fixed form. Its variables map to semantic shifts (what the system says), not just visual ones. Recipe C at desktop redirects to a different recipe entirely: the transformation is a persona handoff, not a layout reflow.
  2. Physical-to-digital continuity: DTPR is unique in that the same brand system must operate at 10m outdoor viewing distance and 320px mobile. The hexagon unit's scale-agnostic formula (p.18) is what makes this coherent — the R formula is the bridge between physical and digital specs.
  3. Chartreuse as the constant: The chartreuse CTA accent is present at every breakpoint and in every recipe because it is the action-signal in the system. Residents tap it. Deployers click it. Developers see it as the selected state. It never disappears — even at minimum survival breakpoints.
  4. QR as the inflection point: The moment a citizen scans a QR code, the system switches from Recipe A (the physical sign's recipe) to Recipe C (the mobile landing's recipe). This deliberate recipe switch at the QR gateway is the most critical interaction handoff in the system and must be tested in all deployment scenarios.
