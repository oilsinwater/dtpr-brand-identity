Sprint Recipes: Three Deployment Paths

A Sprint Recipe is formed by selecting exactly one option from each Morphological Box. The three recipes below define the canonical design paths for DTPR — each tuned to a distinct audience, deployment context, and transparency register.

How to Read a Recipe

Each recipe specifies coordinates across the three boxes as Form[n], Process[n], and Semantic[n]. Option letters map to columns in each table: A = flat/minimal, B = speculative/maker, C = human/ecosystem, D = regulated/standard. These are not rigid rules — they are starting constraints that a design sprint collapses into a single coherent system.

The three recipes are not exhaustive. They represent the three primary deployment tracks identified in the DTPR consolidation plan and the Assembly Day proto-personas. Additional recipes can be composed from the tables as deployment contexts evolve.

FVS Alignment Note

In Lorenz's Flexible Visual Systems, the tool for managing a system's combinatorial variables is the morphological box (Zwicky/Gerstner, pp. 52–53) — "a way to limit your options and keep an overview about them at the same time." Our three boxes are one such morphological box split across three axes; a Sprint Recipe is exactly Lorenz's selection step (locking one value per variable).

  1. Form Box (p.13) → form-based vocabulary: Shape, grid, color, and type — the parameters that, in FVS terms, achieve identification through a distinctive form. These stay stable within a recipe.
  2. Process Box (p.14) → transformation-based vocabulary: The rules for how elements respond to context (viewport, distance, interaction). In FVS terms, identification through a distinctive process. Each recipe defines a distinct transformation envelope.
  3. Semantic Box (p.15) → communication problem + visual-language taxonomy: Deployment track, audience register, reference type, and feedback mechanism. This maps to Lorenz's communication-problem tetrahedron (Product / Target Group / Competitors / Market) and the iconic-vs-symbolic reference axis — it anchors the system to real-world accountability and gives it direction.
  4. Recipe → CAA execution: The morphological box enumerates; the CAA method (Components → Assets → Application) executes. Once a recipe is locked, all downstream work (Geometric Blueprint → Application Assets → Responsive Specs) inherits its constraints. Switching recipes requires revisiting the blueprint.
